# Quiz System

**Feature**: Pre-generated quiz serving with two-level MEMORIZE spaced repetition
**Project**: Learning App

---

## Problem Statement

1. Users learn lessons across many topics but naturally forget material over time. Without adaptive quizzing, reviews either repeat what users already know or miss what they have forgotten.
2. The system needs to track recall at two levels — topic and lesson — so quizzes always target the user's weakest area.
3. Quiz questions are pre-generated offline (not at runtime), so the web app needs to select from a question bank intelligently.

---

## Proposed Solution

Adapt the quiz system from `coach_weight` with these key changes:

- **Two-level MEMORIZE**: Topic > Lesson (coach_weight only had topic-level). Both levels independently track forgetting rates and recall probabilities.
- **Pre-generated question bank**: `QuizQuestion` table with flat columns (coach_weight used a `quiz_data` JSON column).
- **User scope selection**: User can choose all topics, a specific topic, or a specific lesson.
- **Answer format**: Always a list (`["B"]` or `["A", "C"]`), stored as `user_answer` on QuizLog.
- **No "introduced" concept**: All lessons with questions in the bank are eligible.
- **General domain**: Topics and lessons are user-defined, not hardcoded health topics.
- **Quiz types**: `recall`, `understanding`, `application`, `analysis` (coach_weight used `understanding`, `application`, `scenario`, `reflection`).

### What Changes

| Component | Current (coach_weight) | Proposed (learning) |
|-----------|----------------------|---------------------|
| Selection hierarchy | Topic only (principle = least-recently-quizzed) | Topic > Lesson (both use MEMORIZE m(t)) |
| Quiz question storage | `quiz_data` JSON column | Flat columns on `QuizQuestion` |
| Answer format | `"A"` string or `"A,C"` comma-separated | `["A"]` or `["A", "C"]` JSON list |
| User answer storage | Not stored on QuizLog | `user_answer` column on QuizLog |
| Quiz types | understanding/application/scenario/reflection | recall/understanding/application/analysis |
| Scope filtering | None (always auto-select) | Optional `topic_id` / `lesson_id` query params |
| "Introduced" filtering | Filters to introduced principles | No filtering — all lessons eligible |
| Initial forgetting rates | Per-topic hardcoded (0.20–0.40) | Default 0.3 for all, overridable via config |
| Domain | 13 health topics, 131 principles | User-defined topics and lessons (general) |
| `last_asked_at` | On QuizQuestion (global) | Removed — per-user tracking via QuizLog |

## Implementation Plan

### Key Workflow

**To Delete:** None
**To Update:** None
**To Add New:**

Two main flows (see `docs/technical/system_pipelines.md` for full diagrams):

```
Quiz Selection:
  User selects scope → GET /api/quiz/next
    → QuizSelector: resolve scope
    → select weakest topic (MEMORIZE m(t))
    → select weakest lesson in topic (MEMORIZE m(t))
    → select quiz type (rotation)
    → SELECT QuizQuestion from bank (dedup)
    → INSERT QuizLog (pending)
    → Return question + options

Quiz Answer:
  User selects answer → POST /api/quiz/{id}/answer { answer: ["B"] }
    → AnswerService: compare answer to correct_options
    → Store user_answer on QuizLog
    → MemoryService: update BOTH UserTopicMemory + UserLessonMemory
    → Return result + quiz_learnt + all 4 explanations + quiz_take_away
```

```sql
CREATE TABLE quiz_questions (
    id SERIAL PRIMARY KEY,
    topic_id VARCHAR NOT NULL,
    lesson_id INTEGER NOT NULL,
    lesson_filename VARCHAR NOT NULL,
    quiz_type VARCHAR NOT NULL,
    question TEXT NOT NULL,
    quiz_learnt TEXT NOT NULL,
    option_a TEXT NOT NULL,
    option_b TEXT NOT NULL,
    option_c TEXT NOT NULL,
    option_d TEXT NOT NULL,
    correct_options JSONB NOT NULL,
    response_to_user_option_a TEXT NOT NULL,
    response_to_user_option_b TEXT NOT NULL,
    response_to_user_option_c TEXT NOT NULL,
    response_to_user_option_d TEXT NOT NULL,
    quiz_take_away TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_qq_topic_lesson_type ON quiz_questions(topic_id, lesson_id, quiz_type);
CREATE INDEX idx_qq_lesson_type ON quiz_questions(lesson_id, quiz_type);

CREATE TABLE quiz_logs (
    id SERIAL PRIMARY KEY,
    username VARCHAR NOT NULL REFERENCES users(username),
    quiz_question_id INTEGER NOT NULL REFERENCES quiz_questions(id),
    topic_id VARCHAR NOT NULL,
    lesson_id INTEGER NOT NULL,
    quiz_type VARCHAR NOT NULL,
    user_answer JSONB,
    assessment_result VARCHAR,
    created_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_ql_user_topic ON quiz_logs(username, topic_id);
CREATE INDEX idx_ql_user_lesson ON quiz_logs(username, lesson_id);
CREATE INDEX idx_ql_user_created ON quiz_logs(username, created_at);

CREATE TABLE user_topic_memory (
    id SERIAL PRIMARY KEY,
    username VARCHAR NOT NULL REFERENCES users(username),
    topic_id VARCHAR NOT NULL,
    forgetting_rate FLOAT NOT NULL CHECK (forgetting_rate > 0),
    last_review_at TIMESTAMP,
    last_review_quiz_count INTEGER NOT NULL DEFAULT 0,
    review_count INTEGER NOT NULL DEFAULT 0,
    correct_count INTEGER NOT NULL DEFAULT 0,
    UNIQUE(username, topic_id)
);

CREATE TABLE user_lesson_memory (
    id SERIAL PRIMARY KEY,
    username VARCHAR NOT NULL REFERENCES users(username),
    topic_id VARCHAR NOT NULL,
    lesson_id INTEGER NOT NULL,
    forgetting_rate FLOAT NOT NULL CHECK (forgetting_rate > 0),
    last_review_at TIMESTAMP,
    last_review_quiz_count INTEGER NOT NULL DEFAULT 0,
    review_count INTEGER NOT NULL DEFAULT 0,
    correct_count INTEGER NOT NULL DEFAULT 0,
    UNIQUE(username, lesson_id)
);
CREATE INDEX idx_ulm_user_topic ON user_lesson_memory(username, topic_id);
```
